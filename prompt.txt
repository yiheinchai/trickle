/loop 30min REAL-WORLD TESTING WORKFLOW — This is the most valuable type of work.
  The pattern that produces the best results:
  1. Pick a REAL codebase (user's own code, or clone an open-source project)
  2. Run trickle against it: `trickle run python script.py` or `%load_ext trickle` in a notebook
  3. Look at what trickle actually outputs — are the types right? Are values useful? Is the error mode helpful?
  4. Find bugs and gaps through real usage (e.g., "union types show as unknown[]", "error snapshots only show on error line not assignment lines", "autocomplete suggests Tensor methods for str variables")
  5. Fix each issue, test it works, publish and push

  Recent examples of high-value real-world-testing fixes:
  - Error snapshots captured IPython builtins instead of user variables (list comprehension scope bug)
  - Error hints stacked on error line instead of showing on each variable's assignment line
  - union types rendered as "unknown[]" because Python uses "members" not "elements"
  - Autocomplete/highlighting leaked across function scopes
  - Script error snapshots didn't work (importing notebook.py failed in script mode)
  - Line numbers were wrong due to AST unparse reformatting
  - Serializing lists of tensors was slow (str() prints all tensor values)

  These are the bugs that ONLY get found through real-world usage. Synthetic tests miss them.
  When you find and fix these issues, update usecases/ docs and ML_DEBUGGING.md.
