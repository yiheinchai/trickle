# Python Developer: Add Types to Untyped Code Without Writing Them

You have a Python codebase with no type annotations. Instead of manually adding types to hundreds of functions, trickle observes your code at runtime and generates `.pyi` stub files — your IDE immediately gets autocomplete and type checking.

## Install

```bash
pip install trickle-observe
npm install -g trickle-cli      # optional, for CLI commands
```

## Quick Start

### Option A: One import

Add one line to the top of your script:

```python
import trickle.auto
```

Run your script normally:

```bash
python app.py
```

When it finishes, `.pyi` files appear next to your source files with full type stubs. Your IDE picks them up immediately.

### Option B: CLI wrapper (no code changes)

```bash
trickle run python app.py
```

Same result, no import needed.

### Option C: Module runner (no CLI needed)

```bash
TRICKLE_LOCAL=1 python -m trickle app.py
```

## Use Case 1: Generate Type Stubs for an Entire Project

```bash
trickle run python -m pytest tests/
```

Running your test suite captures types for every function that gets called. Then generate stubs:

```bash
trickle stubs src/ --dry-run   # preview what would be generated
trickle stubs src/              # generate .pyi files
```

Result:
```
src/
├── services/
│   ├── user.py
│   ├── user.pyi          ← generated
│   ├── payment.py
│   └── payment.pyi       ← generated
├── models/
│   ├── order.py
│   └── order.pyi         ← generated
```

Each `.pyi` file contains TypedDict definitions and function signatures:

```python
# user.pyi (auto-generated by trickle)
from typing import TypedDict, List, Optional

class CreateUserInput(TypedDict):
    name: str
    email: str
    role: Optional[str]

class UserResponse(TypedDict):
    id: int
    name: str
    email: str
    created_at: str

def create_user(data: CreateUserInput) -> UserResponse: ...
def get_user(user_id: int) -> UserResponse: ...
def list_users(limit: int = ..., offset: int = ...) -> List[UserResponse]: ...
```

## Use Case 2: Annotate Source Files Directly

Instead of generating separate `.pyi` files, inject annotations into your source:

```bash
trickle annotate src/services/user.py --dry-run   # preview
trickle annotate src/services/user.py              # apply
```

Before:
```python
def create_user(data):
    ...
```

After:
```python
def create_user(data: CreateUserInput) -> UserResponse:
    ...
```

## Use Case 3: FastAPI / Flask / Django API Types

Trickle auto-detects your framework and captures request/response types:

```bash
# FastAPI
trickle run uvicorn app:app

# Flask
trickle run python app.py

# Django
trickle run python manage.py runserver
```

Hit some endpoints, then:

```bash
# Python TypedDict stubs
trickle codegen --python -o types.pyi

# Pydantic models
trickle codegen --pydantic -o models.py
```

Generated Pydantic models:

```python
from pydantic import BaseModel
from typing import List, Optional

class User(BaseModel):
    id: int
    name: str
    email: str
    created_at: str

class CreateUserRequest(BaseModel):
    name: str
    email: str
    role: Optional[str] = None
```

## Use Case 4: Pydantic / Dataclass / NamedTuple Inline Values

When you use structured types for config or data models, trickle shows actual field **values** inline — not just the class name:

```python
from pydantic import BaseModel
import dataclasses
from typing import NamedTuple

class AppConfig(BaseModel):
    host: str = "localhost"
    port: int = 8080
    debug: bool = False
    workers: int = 4

@dataclasses.dataclass
class DBConfig:
    url: str = "postgresql://localhost/mydb"
    pool_size: int = 10
    timeout: float = 30.0

class CacheConfig(NamedTuple):
    backend: str
    ttl: int
    max_size: int

app_cfg = AppConfig(port=9000, workers=8)
# → AppConfig(host="localhost", port=9000, debug=False, workers=8)

db_cfg = DBConfig(pool_size=20)
# → DBConfig(url="postgresql://localhost...", pool_size=20, timeout=30.0)

cache_cfg = CacheConfig(backend="redis", ttl=300, max_size=1000)
# → CacheConfig(backend="redis", ttl=300, max_size=1000)
```

Works for all three: Pydantic v1/v2, `@dataclass`, and `NamedTuple`. Hover to see full field types and the complete value as JSON.

## Use Case 5: Observe Specific Modules

```python
from trickle import observe, observe_fn

# Observe all functions in a module
import my_utils
observed_utils = observe(my_utils)

# Observe a single function
@observe_fn
def process_data(records):
    ...
```

Then inspect from the CLI:

```bash
trickle functions                    # list all observed functions
trickle types process_data           # see captured types
```

## Use Case 5: Zero-Code Observation via Import Hook

No decorators, no wrapping. Just run with trickle:

```bash
trickle run python app.py
```

Trickle patches the import system. Every function in every user module (not stdlib, not third-party) gets observed automatically. The first time each function is called, its argument and return types are captured.

Control what gets observed:

```bash
# Only observe specific modules
TRICKLE_OBSERVE_INCLUDE=services,models trickle run python app.py

# Skip noisy modules
TRICKLE_OBSERVE_EXCLUDE=tests,migrations trickle run python app.py
```

## Use Case 6: Variable Tracing in Scripts

Trickle can also trace every variable assignment, not just function signatures:

```bash
trickle run python train.py
```

Then:
```bash
trickle vars                    # all variables with types and sample values
trickle vars --file utils.py    # filter by file
trickle vars --json             # raw JSON output
```

## Use Case 8: Auto-Typing with `trickle.auto`

The simplest approach — one import does everything:

```python
import trickle.auto

# Your code runs normally
# When it finishes, .pyi files are generated
```

Environment variables to control behavior:

| Variable | Description |
|---|---|
| `TRICKLE_INJECT=1` | Inject types directly into source `.py` files |
| `TRICKLE_COVERAGE=1` | Print coverage report (% of functions typed) |
| `TRICKLE_SUMMARY=1` | Print summary of captured types |

## How It Works

1. **Import hook** (`sys.meta_path`) — intercepts `import` statements and wraps every function in user modules with a lightweight observer
2. **AST transformation** — for the entry file (which is already executing when the hook installs), trickle rewrites the AST to add tracing calls
3. **Type inference** — when a function is called, trickle inspects argument and return values to build a type tree (handles dicts, lists, classes, tensors, etc.)
4. **Deduplication** — same type signature is only recorded once per function
5. **Output** — writes to `.trickle/observations.jsonl` (function types) and `.trickle/variables.jsonl` (variable assignments)

Only your code is instrumented — stdlib, site-packages, and popular libraries (torch, numpy, pandas, etc.) are always skipped.
